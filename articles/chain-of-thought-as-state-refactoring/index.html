Chain of Thought as State Refactoring

Jul 10, 2026

The usual explanation for why chain-of-thought reasoning works is that it gives the model a chance to do more computation. But computation to do what exactly?

To answer that, we have to start with a different view of text.

Text as State Machines

Any text is composed of lots of things going through different stages. If you extract the ideas, items, concepts, actors - anything - and follow them, you notice how they change state. Days go by, people go places, change mood, variables in programs get mutated, ideas get developed.

Basically, much of the information held in a text can be viewed as a list of state machines, organised hierarchically. I'm using "state machine" loosely here, not formal automata.

Sometimes the state transitions are explicit. In an itinerary - "Day 1: leave hotel at 9:30, ...", "Day 2: ... visit Colosseum..." - the day is a very explicit context; it's literally the header. The state itself is written in the text.

But most of the time there is no explicit transition at all. "John put the key in his pocket and left." The key is now with John, and John himself left.

And it gets worse. The previous example at least named the "key" concept explicitly, but many times texts silently refer to vague abstract concepts like "the great man theory" in Tolstoy's "War and Peace". The state machine that would track this concept exists only at some higher level of understanding, and is never written down.

Some texts even have concepts that the author himself didn't intend to add consciously, but a clever reader might pick them up and track them.

To understand an existing text, the reader - human or LLM - must identify and track all these state machines, even from highly implicit or vague prose.

In a way, understanding means reconstructing a computation someone else already ran. The author had the ideas, worked out how they connect, and serialised the result into text. The reader's job is to run it again, and get the same mental model.

For ideal understanding, everything could be made explicit, and the tracking would become trivial. But then you get a legal text, or a patent application.

State Machines in LLMs

The details of how LLMs represent state machines shape chain-of-thought.

I won't go into the details of the transformer architecture. What matters for this argument is that the architecture is split into layers. The processing of each token accumulates information from every layer, and information only flows up. When the LLM processes the next token, its lower layers can't read anything from the higher layers of the past tokens.

This means that a fundamental limitation of LLMs is that when a state machine lives very high in the stack and has an unclear state, the LLM has only a few layers above with knowledge that can help clarify it, and, as a rule of thumb, the more implicit something is, the higher in the layers it lives.

Refactoring

Nothing illustrates the constraint better than software programs.

Every large codebase has complex interactions that only a few developers understand - so nobody else can touch some areas because they don't know all the corner cases. The only way out is to refactor and make the logic explicit, living in a single place, with an encapsulated state and a clear API. As soon as this is done, everyone is able to understand it and contribute to it, even though the functionality hasn't changed.

Similarly, in an LLM, when a concern lives at layer 75 of a 100-layer model, only the 25 layers above it can contribute. 75% of the weights, and the knowledge stored in them, never get a chance to contribute, because they never see the concern. Even worse, they can contribute with irrelevant or contradictory information.

The solution is pretty much the same: refactor and bring this concept down to a lower level, and it turns out that models can do that easily because they can write text, so all it takes is to verbalise the state machine itself and feed it back as input.

One useful way to view chain-of-thought is models learning to refactor the tricky areas of their internal state by writing it down, and, thus, engaging all the contributors.

We can actually see this in action, because researchers at Anthropic studied what happens internally when a model answers "What is the capital of the state containing Dallas?". There are two dependent state machines here: one resolves Dallas to Texas, the other resolves a state to its capital, and they live at different depths. The Dallas-to-Texas machine finishes somewhere in the middle of the stack, so the capital machine only has the remaining layers to do its work.

When running without reasoning, the model manages in this case, but the pattern is fragile: models often answer both sub-questions correctly in isolation and still fail the combined question. With chain-of-thought, the model instead writes: "Dallas is in Texas. The capital of Texas is Austin." Nothing clever going on here; it just serialises the state.

Reasoning models also write things like "This is an inclusion-exclusion problem" or "This is a system of two linear equations". That classification is the most implicit state there is, a pattern the problem itself never states. Writing the name down matters because the model's knowledge about how to solve such problems is likely keyed on those concrete words. The concern gets named, and the name brings in every contributor that has ever seen that kind of problem.

Optionality and Backtracking

State machines might have multiple plausible states, and language models are built for exactly this, because they keep all options open. For example, during processing, John's key is held as something like 90% in the car, 10% in the house.

But there is a tension if written tokens are treated as hard choices.

Earlier models committed early and then defended the commitment. For example, if "Yes" is the first token of the answer, coherence forced everything after it to back it up.

The simplest solution is to hedge during refactoring and write the state as "the key is in the house or in the car", and not make any decision at that stage.

But there is a more powerful technique which turned out to be very useful for harder problems.

The solution that emerged during reinforcement learning is one more state machine: "thinking mode". The delimiter tokens were given to the model by its training template, but what to do inside them is what emerged during RL.

Basically, models are free to commit to a choice when refactoring an uncertain state machine, and carry on with the processing. If it stops looking good, they write "wait" and take another branch of the last state machine by writing down that variant.

This technique allows the model to explore many variants by implementing a backtracking algorithm and using its entire knowledge for every possibility. Each option can lead to more options if it happens to match some pattern.

This refactoring view of chain-of-thought explains why it is not reliably a debug trace. Models sometimes write reasons that are not the real reasons, and the written trace can understate or misstate what happened inside.

Basically, writing is a promotion of selected state, not a debug trace. The chain gets written to recruit contributors and to trigger more exploration.

Conclusion

Chain-of-thought reasoning is a way for models to extend their understanding process. Implicit, higher-level state machines that don't have a clear resolution are serialised into text, which the entire machinery can then work on. The models found a way to do it while preserving optionality. Chain-of-thought becomes, in a way, a process of continuous refactoring, where the model extracts and names concerns until it reaches a solution.
